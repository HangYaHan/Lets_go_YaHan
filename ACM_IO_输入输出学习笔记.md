# ACM I/O 学习笔记（基于本项目实战）

## 1. 学习目标与方法

- 目标：在面试/笔试/ACM 环境中，快速识别输入协议并写出稳定、格式严格的输出。
- 方法：每次只做一题，保留题面注释，完成后切到下一题。
- 原则：先保证读对，再保证算对，最后保证格式 100% 匹配。

---

## 2. 核心输入模式总览

### 2.1 固定字段（单组）

- 典型：`a b`，输出 `a+b`。
- 写法：`cin >> a >> b; cout << a + b;`
- 坑点：多余空格通常无影响，末尾不要多输出换行或空格（按题意）。

### 2.2 固定字段（混合类型）

- 典型：`int + double + string`。
- 关键：`fixed + setprecision(n)` 控制小数位。
- 结论：
  - `fixed` 后 `setprecision(2)` 表示保留 2 位小数。
  - 不加 `fixed` 时 `setprecision` 常表示有效数字位数。

### 2.3 固定多行 / T 组输入

- 题目给 `T` 就按 `for (i=0; i<T; ++i)` 读。
- 每组输出一行时，建议统一 `<< '\n'`。

### 2.4 T 组 + 组内变长

- 模式：`T` 组；每组先读 `n`，再读 `n` 个值。
- 强化点：嵌套循环与每组状态重置（`sum = 0`）。

### 2.5 EOF 连续读取（定长）

- 模式：组数未知，读到文件结束。
- 模板：`while (cin >> a >> b) { ... }`
- 适用条件：题面明确写了“直到 EOF”，且无 T、无哨兵。

### 2.6 EOF 连续读取（每行不定长）

- 模式：每行若干整数，按行求和。
- 模板：
  - `getline(cin, line)` 保留行边界。
  - `istringstream iss(line); while (iss >> x) sum += x;`
- 空行会得到和 0（因为内层循环不执行）。

### 2.7 哨兵结束

- 模式：遇到特殊值结束，比如 `0 0`。
- 要点：先判断哨兵，再输出。

### 2.8 记录型输入

- 模式：`name score` 这样的字符串+数字组合。
- 建议：数值统计用 `double/long long`，输出按题意保留位数。

### 2.9 矩阵输入 / 多组矩阵（EOF）

- 双层循环是基础。
- 多组 EOF 读法中，每组累加变量必须重置。

### 2.10 图边列表

- 输入：`n m` + `m` 条 `u v`。
- 无向图度数统计：`degree[u]++`, `degree[v]++`。
- `vector<int> degree(n + 1, 0)` 原因：点编号通常是 1..n，避免偏移换算错误。

### 2.11 命令流解析

- 模式：`ADD x / MUL x / PRINT / END`。
- 重点：字符串分支 + 状态变量维护。

### 2.12 一整行文本题

- 统计单词：`istringstream` + `>> word` 自动跳过多空格。
- 单词反转：先切词入容器，再逆序输出，控制单词间单空格。

### 2.13 cin + getline 混用

- 问题：`cin >> n` 后换行还在缓冲区，直接 `getline` 会读到空串。
- 解决：`cin.ignore()` 或更稳健版本：
  - `cin.ignore(numeric_limits<streamsize>::max(), '\n');`

### 2.14 CSV 简化解析

- `getline(iss, field, ',')` 按逗号切字段。
- 能快速完成字段计数、提取等任务。

### 2.15 浮点 EOF / 除零分支

- 浮点格式统一：`fixed << setprecision(2/6)`。
- 除法题要先判断分母为 0。

### 2.16 Case 格式与组间空行

- `Case i: ans` 编号从 1 开始。
- 组间空行控制：
  - 常见写法 `if (i) cout << '\n';`
  - 避免最后多空行。

### 2.17 超长整数相加

- 不能用内置整数，需字符串模拟加法。
- 核心：从末位到首位逐位相加并处理进位。

### 2.18 脏输入鲁棒性

- 先分词，再判断 token 是否整数。
- 使用 `stoi` 时注意异常与“部分转换成功”问题。

### 2.19 日志行解析

- `find('|')` 找分隔位置。
- `substr` 提取字段。
- `unordered_map<string,int>` 做 level 计数。

### 2.20 综合协议输入

- 首字段决定后续读法（INT/FLOAT/LINE）。
- 本质是状态切换与多模式兼容。

## 3. 常见库与函数速记

### 3.1 `iomanip`

- 用途：格式控制。
- 高频：`fixed`, `setprecision`, `setw`, `setfill`, `left`, `right`, `boolalpha`。

### 3.2 `getline`

- `getline(cin, line)`：读整行（默认分隔符换行）。
- `getline(iss, field, ',')`：按自定义分隔符读取字段。

### 3.3 `find / npos / substr`

- `s.find('|')`：找字符位置。
- `string::npos`：未找到。
- `s.substr(pos, len)`：截子串。

### 3.4 `stoi` 与异常

- 正确函数：`stoi`（不是 stdoi）。
- 可能抛：
  - `invalid_argument`：无法解析数字。
  - `out_of_range`：超范围。
- 严格整串为整数：`stoi(token, &pos)` 后检查 `pos == token.size()`。

### 3.5 `static_cast`

- 典型：`static_cast<int>(words.size())`。
- 作用：显式类型转换，避免无符号倒序循环陷阱。

---

## 4. 为什么 `>>` 会“跳过空格”

- `operator>>` 默认有 `skipws` 行为，会先跳过前导空白字符。
- 空白包括空格、TAB、换行等。
- 若要统计空格数量，应该按字符遍历整行，而不是用 `>>` 分词。

## 5. 测试与批改工作流（本项目）

### 5.1 输入数据组织

- 每题 4 组输入，路径：`tests/inputs/题号/1.in ... 4.in`。

### 5.2 批改脚本

- 脚本：`tests/judge.ps1`
- 运行（根目录）：
  - `./tests/judge.ps1 -Problem 01 -Compile`

### 5.3 为什么会出现 `[RAN]`

- 含义：程序已运行，但没有对应 `tests/outputs/.../*.out`，所以无法做 PASS/FAIL 对比。

### 5.4 如何得到 PASS/FAIL

- 准备期望输出（`.out`）。
- 再次运行 judge，脚本比较 got 与 out 一致性。

### 5.5 重定向与管道

- 推荐重定向：`prog < in > got`，最贴近 OJ。
- 管道可用于临时调试，但格式边界处理不如重定向稳定。

## 6. 你实战中踩过/已修正的关键点

1. 组内变长题：必须先读 `n` 再读 `n` 个元素。
2. `reserve` 只分配容量，不会创建元素。
3. `pop_back()` 返回 `void`，不能拿来累加。
4. EOF 题一般每组输出一行，注意换行格式。
5. `judge` 不是系统命令，PowerShell 运行脚本需要 `./` 前缀。
6. `setprecision` 拼写要正确。

---

## 7. 输出格式黄金规则

1. 按题意决定是否输出末尾换行。
2. 单词/字段间空格数量要严格一致。
3. 组间空行要精确控制，不多不少。
4. 浮点输出固定使用 `fixed + setprecision`。

---

## 8. 一份可复用的 I/O 选择表

1. 给定 T 组：`for` 循环。
2. 未给组数且直到 EOF：`while (cin >> ...)`。
3. 按行处理：`getline`。
4. 行内二次解析：`istringstream`。
5. 有哨兵：循环内优先判断哨兵。
6. 混用 `>>` 和 `getline`：中间 `cin.ignore(...)`。

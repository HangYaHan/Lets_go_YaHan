## 二、项目深挖：Beruto 回测系统（16-33）

### 16. Beruto 从 Python v1.0 到 Rust 重写的动机
主答：
Beruto 的核心目标是把“策略实验”做成长期可扩展的工程，而不只是一次性脚本。重写到 Rust，主要是为了三点：第一，类型系统把很多错误提前到编译期；第二，模块边界更清晰，适合后续扩展 CLI、结果存储、策略市场；第三，数据处理和回测循环能在更低运行时开销下保持稳定。当前仓库已经按 `data/strategy/backtest/cli` 拆分，并形成可运行的闭环。

追问答（如果只做局部优化先动哪两块）：
我会先动 `backtest::engine` 和 `data::data_source`。前者是主计算路径，后者是数据正确性的入口，先把“算得对、算得稳”打牢，再谈更多策略和并发。

### 17. 核心模块划分与拆分理由
主答：
模块拆成四层：
- `data`：负责 CSV 加载与字段解析。
- `strategy`：统一策略接口 `Strategy`，策略只关心信号决策。
- `backtest`：统一资金和持仓状态推进，输出净值、回撤、交易次数。
- `cli`：参数解析、执行回测、保存结果、排行榜。

这样拆的好处是“数据、决策、执行、交互”职责分离，后续改任一层时影响范围可控。

追问答（如何防止扩展困难）：
把策略抽象成 trait，不让策略直接操作现金和仓位；策略只发 `Buy/Hold/Sell`，引擎统一处理交易语义。

### 18. 策略接口如何统一
主答：
通过 `Strategy` trait 统一，回测引擎只依赖 trait，不依赖具体策略类型。当前 `BuyAndHold` 和 `Contrarian` 都通过同一个入口被执行。

追问答（新增多因子会不会破坏接口）：
不会。多因子策略只需要在 `on_bar` 内部组合因子逻辑并继续返回 `Signal`。如需更多上下文，可在不破坏现有调用方的前提下扩展输入结构。

### 19. Python 版本性能瓶颈主要在哪
主答：
典型瓶颈在“逐 bar 循环 + 状态更新 + 指标计算”这条热路径。Beruto 的 Rust 实现把这条路径集中在 `run_backtest`，便于持续优化和基准测试。

追问答（如何证明优化有效）：
目前仓库有功能性集成测试，没有基准数据。面试时可以诚实说：已完成工程重构与可测闭环，下一步补 `criterion` 或固定样本耗时对比，形成可复现性能报告。

### 20. Rust 版本最难处理的数据结构
主答：
难点不是“复杂容器”，而是“有状态策略 + 回测状态机”的边界管理：策略要有内部状态（如上一日收盘价），引擎还要保证资金和持仓的合法迁移。

追问答（是否使用 Rc/Arc/RefCell）：
当前主流程是单线程同步，不需要 `Rc/Arc/RefCell`。优先保持简单所有权模型，减少可变共享带来的复杂性。

### 21. 如何处理交易状态时序一致性
主答：
在 `run_backtest` 里明确约束：`shares == 0` 才允许买，`shares > 0` 才允许卖。这样天然避免“先卖后买”或重复买入的非法状态。

追问答（如何避免非法状态）：
把合法状态迁移写死在引擎层，而不是分散到各策略里。策略只负责信号，引擎负责状态机。

### 22. 收益/回撤/交易次数如何计算与验证
主答：
- 收益率：`(final_equity / initial_capital - 1) * 100%`
- 最大回撤：遍历净值曲线，维护滚动峰值并计算当前回撤。
- 交易次数：每次成功买或卖都计数。

验证方式：集成测试校验净值曲线长度、结果非负、回撤范围在 0~100 之间。

追问答（对照实验）：
可用一段小样本手工验算，再和程序输出逐项对比；再用同一数据集做策略 A/B 输出比对，排除指标计算偏差。

### 23. 阈值 sweep 对比怎么设计
主答：
`contrarian` 策略支持 `--buy-drop` 和 `--sell-rise` 参数。做 sweep 时固定数据和初始资金，批量运行不同阈值组合，最后按收益、回撤、交易次数综合比较。

追问答（避免过拟合）：
时间切分训练/验证，且不要只看收益率。至少联合看回撤和交易频率，避免参数只贴合某段历史噪声。

### 24. 输入数据清洗与校验
主答：
`data_source` 里做了列名校验和数值解析报错，缺字段直接报错；`noon_close` 缺失时回退到 `close`。这保证了最低限度的数据可用性和可解释性。

追问答（缺失值/脏数据/错序）：
当前代码主要处理“字段缺失和格式错误”。时间戳错序和重复值建议在加载后增加统一排序与去重步骤，并输出清洗日志。

### 25. Python 与 Rust 结果一致性如何保证
主答：
当前仓库内部通过集成测试保证 Rust 结果稳定，但没有内置 Python 对照脚本。落地做法是引入一份 golden data，用同一输入在两端跑，再对关键指标做容差比对。

追问答（是否有 golden data）：
目前没有显式 golden 测试文件，建议补一组固定样本到 `tests/fixtures`，并把期望输出固化到回归测试。

### 26. 集成测试覆盖关键路径
主答：
已有集成测试覆盖：
- CSV 加载成功。
- BuyAndHold 与 Contrarian 可完整跑通。
- 可配置阈值策略可运行。
- 回测结果关键字段有效。

追问答（最低可接受覆盖）：
先保证“数据加载 -> 策略执行 -> 指标产出 -> 结果结构”这条主链路 100% 可跑，再逐步补边界条件测试。

### 27. 若引入多线程加速，先担心什么
主答：
先担心可重复性和状态隔离。策略通常有内部状态，不能在线程间共享同一个可变实例，否则结果会漂移。

追问答（如何避免共享状态不稳定）：
按任务粒度并行，每个线程独立持有策略和上下文；共享部分只读，结果汇总阶段再合并。

### 28. 支持实盘仿真要改哪些点
主答：
需要从“离线批处理”升级为“事件驱动”：
- 数据层支持流式行情。
- 引擎支持订单生命周期和撮合回报。
- 增加风控、超时、断线重连和状态恢复。

追问答（事件驱动/延迟/容错）：
先定义统一事件总线，再把策略、撮合、风控解耦；关键状态做持久化快照，保证故障后可恢复。

### 29. 最有价值的一次重构
主答：
最有价值的是把“策略逻辑”和“交易状态推进”彻底分离：策略只出信号，引擎统一管资金和仓位。收益是可测性更好、策略扩展成本更低。

追问答（效率与 bug 率变化）：
仓库未固化具体统计值。可给出定性结论：功能迭代时回归范围更可控，新增策略无需改引擎主流程。建议补 issue 统计或迭代数据做量化。

### 30. 结果与预期不一致时的排查路径
主答：
分三层排查：
1) 数据层：CSV 是否完整、字段是否错位。
2) 策略层：每个 bar 的信号是否符合预期。
3) 引擎层：买卖执行与资金变更是否正确。

追问答（先看日志/数据/代码）：
先看数据和关键日志，再看代码。因为很多偏差来自输入问题或参数配置，而不是算法本体。

### 31. 如何让项目更友好给协作者
主答：
优先补三件事：
- 统一运行入口和示例命令。
- 固定样本回归测试。
- 结果文件格式说明（run record 字段含义）。

追问答（文档/示例/测试/CI 先补哪个）：
先补“可运行示例 + 测试”，因为它们直接决定新同学是否能快速建立正确心智模型。

### 32. 业务要“秒级返回回测结果”怎么优化
主答：
分两层：
- 算法层：减少不必要计算、指标增量更新。
- 工程层：并行参数扫描、缓存中间结果、预加载数据。

追问答（算法层 vs 工程层）：
算法层解决复杂度，工程层解决吞吐和延迟，两者要同时推进。先做低风险工程优化，再做算法重构。

### 33. 踩坑与复盘
主答：
典型坑是“策略状态和引擎状态边界不清”导致结果难解释。复盘后统一了责任边界：策略只给信号，引擎统一处理持仓语义。

追问答（是否沉淀规范）：
建议沉淀成模板：新策略必须通过 smoke test，且必须提供参数说明与默认值。

## 三、项目深挖：色温数据采集与训练工具（34-43）

### 34. 如何设计 pics/specs 配对校验规则
主答：
当前 TaskHub 按同一轮次编号保存 `frame_N.jpg` 与 `spectrum_N.csv`，天然形成一一配对。校验规则可以基于“同轮次、同目录、文件同时存在”实现。

追问答（失败如何友好反馈）：
在控制台输出“具体轮次 + 缺失类型（frame/spec）+ 路径”，并提供一键导出失败清单。

### 35. QThread + Signal 设计如何避免 UI 卡死
主答：
当前实现以 Qt 事件循环 + `QTimer` 驱动任务，不用阻塞式 while 循环；相机与光谱更新都通过回调推进，UI 不会被长时间占用。

追问答（紧急停止如何资源释放）：
`Stop` 会停止计时器；相机和光谱面板提供显式 close/shutdown，退出时统一保存设置并释放设备句柄。

### 36. ROI/BG ROI 提取可重复性
主答：
当前代码里已固定图像灰度转换公式（0.299/0.587/0.114）和直方图计算流程，保证同输入同输出。若扩展 ROI/BG ROI，建议把 ROI 坐标与预处理参数一起落盘。

追问答（不同光照是否动态阈值）：
建议支持动态阈值，但默认保留固定阈值基线，避免引入不可解释波动。

### 37. 背景归一化对模型指标影响如何评估
主答：
用单变量 A/B：只切换“是否做背景归一化”，保持数据切分、模型结构、训练轮次一致，对比 RMSE/MAE/R2 变化。

追问答（如何隔离单变量影响）：
固定随机种子和训练集划分，其他参数不变，多次重复后取均值和方差。

### 38. 为什么做 spectra/cct 双模式训练
主答：
从工程角度，系统同时支持“光谱数据采集”和“目标色温导向控制（Indoor CCT）”。双模式有助于分别学习“精细光谱形状”和“色温目标达成”的不同任务。

追问答（数据依赖和误差差异）：
spectra 模式更依赖高维曲线质量；cct 模式更依赖目标定义和映射稳定性，通常误差分布更集中在目标附近偏差。

### 39. 异常样本和离群值怎么处理
主答：
先分层：采集失败（文件缺失）、设备异常（mock 数据）、统计离群（数值异常）。前两类直接剔除并记录日志，后者做阈值或鲁棒统计处理。

追问答（剔除与修正边界）：
当样本违反物理约束或采集链路失败时剔除；可解释且可校正的偏差才修正。

### 40. RMSE/R2/MAE/MAPE 在场景中的意义
主答：
- RMSE：放大大误差，适合看极端偏差风险。
- MAE：平均绝对误差，直观稳健。
- R2：解释度，衡量趋势拟合能力。
- MAPE：相对误差，便于业务理解百分比偏差。

追问答（指标矛盾怎么判断）：
看业务容忍度。如果极端误差风险高，优先 RMSE；如果业务看平均稳定性，优先 MAE/MAPE。

### 41. 模型与数据摘要落盘的版本管理
主答：
当前系统已有 `settings.json` 和任务输出目录。可进一步落地为“run_id 目录”：保存参数快照、输入数据摘要、结果指标和日志，做到可追溯。

追问答（如何快速定位一次训练）：
统一命名 `run_id + 时间戳`，并在索引表中记录数据路径、参数哈希、指标摘要。

### 42. 单机工具服务化改造
主答：
拆成三层：
- API 层：任务创建、状态查询、结果下载。
- 调度层：任务队列和 worker。
- 资源层：设备访问隔离与并发配额。

追问答（队列/鉴权/并发隔离）：
队列用优先级任务；鉴权用 token + 项目配额；设备类资源采用独占锁与租约机制。

### 43. 工程价值大于算法价值的点
主答：
最大工程价值是把“相机、光谱、LED 串口、任务循环、命令控制”集成为一个稳定工作台。它直接提升数据采集效率和可重复性，是后续算法效果的地基。

追问答（再做一次先改什么）：
第一优先级补“任务级元数据与失败追踪”，让每轮采集都可追溯、可复现、可审计。

## 四、项目深挖：BiliBili 文件处理工具（44-51）

### 44. 目录扫描与元数据解析如何保证稳定
主答：
流程是“扫描目录 -> 仅保留数字文件夹 -> 校验 `.m4s`/`.json` 数量 -> 解析标题”。通过前置校验把异常目录尽早过滤，保证主流程稳定。

追问答（软链接/权限/非法名）：
当前实现主要覆盖常规目录。权限和软链接异常需要在 `directory_iterator` 外围补 try-catch 和错误分级日志。

### 45. `.m4s -> .mp4 -> .mp3` 如何做失败重试与幂等
主答：
当前链路是串行批处理：先把目标 m4s 头部跳过 9 字节写成 mp4，再调用 ffmpeg 转 mp3，最后清理临时 mp4。失败会计数并继续处理下一文件。

追问答（断点续跑设计）：
建议落地任务清单（pending/running/success/failed）并记录输出校验，重跑时跳过已成功任务即可实现幂等。

### 46. 中文文件名编码问题
主答：
程序使用 UTF-16 控制台输出和 `wstring` 路径，JSON 标题按 UTF-8 解码后转宽字符，写文件时 `_wfopen` 兼容中文路径。

追问答（UTF-8 与系统编码冲突）：
统一内部编码为 UTF-8，边界层按平台转换；Windows 路径和控制台显示统一走宽字符接口。

### 47. FileManager/JsonParser/MediaProcessor 如何拆
主答：
- `FileManager`：目录与扩展名筛选。
- `JsonParser`：读取 `entry.json` 并提取标题。
- `MediaProcessor`：判定目标 m4s、执行 m4s->mp4 转换。
- `main`：流程编排与统计输出。

追问答（边界错会怎样）：
一旦业务逻辑混入基础工具层，后续改一个环节会牵连全局，测试和排障成本都会明显上升。

### 48. CMake 关键改进
主答：
使用 `GLOB_RECURSE` 自动收集源码，统一 C++20 标准，设置运行产物目录，并在 MinGW 下启用静态链接参数，降低部署时运行库依赖问题。

追问答（新同学 5 分钟跑起来）：
提供一键脚本：检查 ffmpeg、配置构建目录、执行 cmake build，并输出运行前目录检查清单。

### 49. 提升批量转换吞吐量从哪入手
主答：
先拆分 I/O 与转码阶段：目录扫描和 m4s 修复可并发，ffmpeg 转码可做进程池限流。避免所有任务串行等待。

追问答（多进程 vs 线程池）：
转码属于外部进程调用，优先多进程任务调度；线程池更适合轻量 I/O 和元数据处理。

### 50. 如何设计日志方便定位
主答：
建议每个文件夹输出结构化日志：输入目录、标题、关键文件数量、是否成功、失败原因、耗时。当前已有控制台进度日志，可继续升级到文件日志。

追问答（日志级别和关键字段）：
级别分 INFO/WARN/ERROR，字段至少包含任务 id、目录、阶段、错误码、耗时。

### 51. 做成后端服务的 API 设计
主答：
建议最小 API 集：
- 创建任务：提交目录或上传清单。
- 查询任务：返回进度、阶段、失败原因。
- 下载结果：返回 mp3 列表或打包文件。

追问答（鉴权/状态查询/配额）：
鉴权用 token；状态查询按任务 id 轮询或 websocket 推送；配额按用户并发任务数和日处理量限制。

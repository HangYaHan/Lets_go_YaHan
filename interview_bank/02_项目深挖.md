# 二、项目深挖：Beruto 回测系统（18题）

16. 题目：请你完整讲一下 Beruto 从 Python v1.0 到 Rust 重写的动机。
考察点：重构理由、技术决策闭环。
追问方向：如果只允许做局部优化而非重写，你会选哪 2 个模块先动手？

17. 题目：回测系统的核心模块划分是什么？为什么这样拆？
考察点：模块化设计能力、边界抽象。
追问方向：模块之间如何解耦，接口如何防止未来扩展困难？

18. 题目：策略接口是如何统一的？
考察点：接口设计、可扩展性。
追问方向：如果新增多因子策略，现有接口会不会被破坏？

19. 题目：Python 版本的性能瓶颈主要出现在哪里？
考察点：性能定位能力、测量意识。
追问方向：你是如何证明优化有效，而不是“感觉更快”？

20. 题目：Rust 版本中最难处理的数据结构是什么？
考察点：Rust 所有权理解、复杂状态管理能力。
追问方向：你是否使用了 Rc/Arc/RefCell，为什么？

21. 题目：你如何处理交易状态的时序一致性问题？
考察点：状态机建模、时序正确性。
追问方向：如何避免“先卖后买”这类非法状态？

22. 题目：收益、回撤、交易次数这些指标是如何计算与验证的？
考察点：指标定义清晰性、验证方法。
追问方向：你做过哪些对照实验来排除计算错误？

23. 题目：阈值 sweep 对比是怎么设计的？
考察点：实验设计能力、自动化意识。
追问方向：如何避免参数搜索导致过拟合历史数据？

24. 题目：你如何设计回测输入数据的清洗与校验？
考察点：数据质量控制、异常处理。
追问方向：遇到缺失值、脏数据、时间戳错序怎么处理？

25. 题目：你如何保证 Python 与 Rust 两个版本结果一致？
考察点：跨实现一致性验证。
追问方向：是否有 golden data 或回归样本集？

26. 题目：集成测试覆盖了哪些关键路径？
考察点：测试设计完整性。
追问方向：你如何定义“最低可接受覆盖范围”？

27. 题目：如果引入多线程回测加速，你首先担心什么问题？
考察点：并发风险识别能力。
追问方向：如何避免共享状态导致结果不稳定？

28. 题目：回测系统如果要支持实盘仿真，架构上要改哪些点？
考察点：从离线到在线的迁移思维。
追问方向：事件驱动、延迟控制、容错策略怎么设计？

29. 题目：你在这个项目中做过最有价值的一次重构是什么？
考察点：重构判断与收益评估。
追问方向：重构前后开发效率和 bug 率变化如何？

30. 题目：如果策略结果和预期不一致，你的排查路径是什么？
考察点：Debug 方法论、问题分层能力。
追问方向：你会优先看日志、数据还是代码逻辑？为什么？

31. 题目：你如何让这个项目对未来协作者更友好？
考察点：可维护性、协作意识。
追问方向：文档、示例、测试、CI 你会先补哪个？

32. 题目：如果业务侧希望“秒级返回回测结果”，你会如何优化？
考察点：性能与架构优化思路。
追问方向：哪些优化是算法层，哪些是工程层？

33. 题目：请讲一个 Beruto 项目中你踩过的坑和复盘结论。
考察点：真实性、复盘深度。
追问方向：这个复盘是否沉淀成了规范或模板？

42. 题目：如果要把工具改为多人协作服务化平台，你怎么改架构？

43. 题目：这个项目中你认为“工程价值”大于“算法价值”的点是什么？
考察点：价值判断、系统思维。
追问方向：如果让你再做一次，第一优先级会改什么？
